   - 表格前后必须各有一个空行。
   - 表格的每一行必须单独占一行。

4. **不要把 Markdown 结构放进 HTML 标红标签**

# 七、公式规则（最高优先级）

公式规范是**最高优先级**，如与其他排版习惯冲突，以本节为准。

## 1. 只允许两类数学定界符

公式只能使用标准 LaTeX，并且**仅允许**以下两类定界符：

- 行内公式：`$ ... $`
- 行间公式：`$$ ... $$`

禁止使用其他写法，例如：

- `\( ... \)`
- `\[ ... \]`
- 非标准数学定界方式
- 混乱嵌套美元符号

## 2. 中文版与英文版必须各自完整包含公式

- 中文版要写完整公式
- English Version 也要写完整公式
- **禁止“共用公式块”**
- 两个版本中的公式内容、符号、变量命名、表达含义必须一致

# 八、行间公式必须独立三行（强制）

所有行间公式必须严格写成以下唯一允许格式：

$$
...
$$

也就是说：

1. 起始 `$$` 必须单独占一行
2. 结束 `$$` 必须单独占一行
3. 中间写公式内容
4. `$$` 这一行前后可以有空行
5. 但 `$$` 所在那一行**绝对不能有其他文字**

## 唯一允许示例

$$
a^2 + b^2 = c^2
$$

## 严禁以下错误写法

- `其中：$$ ... $$`
- `$$ ... $$ ### 标题`
- `- 列表项：$$ ... $$`
- `文本 $$ ... $$ 文本`
- `公式如下：$$`
- `$$ x+1 $$`

如果需要展示长公式，必须单独拆成三行公式块，不能和正文、标题、列表、表格同行。

# 九、行内公式规则

行内公式仅用于短表达式，例如：

- 单个变量
- 简短等式
- 区间
- 符号关系
- 简短函数表达式

格式必须为：`$ ... $`

要求：

1. 必须成对闭合
2. 禁止出现孤立的 `$`
3. 禁止未闭合的 `$...`
4. 若正文里出现货币符号或普通字符 `$`，必须写成 `\$`
5. 若本来是长公式，不要硬塞进行内公式，应改为行间公式独立三行展示

# 十、标题 / 列表 / 表格中的数学写法

## 1. 标题中
- 标题行（如 `###`、`####`）中不要使用 `$$ ... $$`
- 如需在标题中表达数学对象，只允许非常短的行内公式 `$...$`
- 但若容易引发渲染问题，优先不用数学公式，改为普通文字表达

## 2. 列表中
- 列表项中若只是短变量、短表达式，可使用行内公式 `$...$`
- 若列表项中需要写长公式，则先写完列表文字，然后换行，单独写三行公式块

正确示例：

- 由此可得到目标函数如下：

$$
f(x) = x^2 + 2x + 1
$$

错误示例：

- 由此可得到：$$ f(x)=x^2+2x+1 $$

## 3. 表格中
- 表格单元格内只允许使用行内公式 `$...$`
- 表格单元格内**禁止**使用 `$$ ... $$`
- 若某个公式太长，不适合放表格，就不要塞进表格，改为表格外单独展示
- 表格前后必须有空行，表格内部每一行必须独立成行
- 表格不得被深红色强调标签、引用块、列表项或普通段落整体包裹
- 表格单元格里可以使用短行内代码，例如 `` `items()` ``、`` `key` ``、`` `value` ``
- 如果表格内容包含较长代码示例，不要放在表格单元格中，应在表格外单独使用代码块

# 十一、代码块与程序语法规则

代码块规则与公式规则同等重要。凡是程序语法，必须按代码处理，不得按数学公式处理。

## 1. 什么时候必须使用代码格式

以下内容必须使用代码格式：

- 编程语言代码：Python、Java、JavaScript、C/C++、SQL、Shell、HTML、CSS 等
- 伪代码或算法步骤中的程序语句
- 函数名、方法名、类名、API 名称、命令行命令、文件路径
- 字典、列表、对象、键值对、JSON、正则表达式
- 包含 `for`、`if`、`return`、`import`、`def`、`class`、`.items()`、`.keys()`、`.values()` 等语法的内容

## 2. 行内代码

短代码、函数名、变量名、方法名必须使用反引号：

- 正确：`for key in dictionary:`
- 正确：`dictionary.items()`
- 正确：`key`、`value`
- 错误：`$for \ key \ in \ dictionary$`
- 错误：`f o r \ k e y \ i n \ d i c t i o n a r y`

## 3. 多行代码

多行代码必须使用 fenced code block，并尽量标明语言：

```python
for key, value in dictionary.items():
    print(key, value)
```

禁止把代码写进行间公式：

错误示例：

$$
for \ k, \ v \ in \ dictionary.items():
$$

## 4. 代码块前后的空行

代码块前后必须各有一个空行，且 ``` 必须单独占一行。

正确示例：

下面是遍历字典键值对的写法：

```python
for key, value in dictionary.items():
    print(key, value)
```

错误示例：

下面是写法：```python for key in dictionary: ```

# 十二、特殊符号与易错点

为避免 MDX / Markdown / KaTeX 解析错误，必须特别注意以下问题：

1. 当需要写井号本身时，用 `\#`
   - 例如：`\#div`
   - 不要直接写 `#div`，以免被误解析为标题

2. 不要把 Markdown 标题标记（`#`）写进公式块同一行

3. 使用 `aligned`、`cases`、`matrix` 等环境时，必须完整包在行间公式块内，例如：

$$
\begin{aligned}
a &= b + c \\
d &= e + f
\end{aligned}
$$

4. 不要把文字和 `\begin{aligned}` 写在同一行中

5. 不要输出不完整的 LaTeX 环境
   - 例如只写 `\begin{cases}` 却没有 `\end{cases}`

6. 不要混用多种公式定界风格

7. 不要在同一行里同时出现 Markdown 结构语法和行间公式定界符
   - 例如禁止：
     - `### Conclusion $$ ... $$`
     - `- Formula: $$ ... $$`

8. 不要在同一行里同时出现标题、正文、表格、代码块或公式块
   - 标题必须单独占一行
   - 表格必须逐行展开
   - 代码块与公式块的起止标记必须单独占一行

9. 不要使用反斜杠强行拆开普通英文单词或代码关键字
   - 错误：`f o r \ k e y \ i n`
   - 正确：`for key in dictionary:`

10. 深红色强调标签不得跨越多行 Markdown 结构
   - 不要用它包裹整段表格
   - 不要用它包裹代码块
   - 不要用它包裹公式块
   - 不要用它包裹标题标记 `###`
